## 安全与边界

### 当前安全模型（现实约束）

`codex-cli` 通过模型生成 patch 并在本地应用，因此它的安全边界主要依赖：

- 规则文本（注入 system prompt）
- 受保护文件/目录硬过滤
- `--yes`（是否允许提交/推送）作为“最后一道闸”
- prompt-based 安全审计（软提示，不替代真实扫描）

如果你需要更强的保障，建议在仓库 CI 中叠加真实的 SAST/secret scan，并在合并前强制通过。

### 关键机制

**1) 受保护文件过滤（硬过滤）**

- 默认保护：锁文件、关键配置文件、`.env` 等
- 可扩展：`CODEX_PROTECTED_FILES=...`

目的：减少对高风险/高影响文件的自动修改。

**2) Dry-Run 默认安全**

未传 `--yes` 时：

- 仍会生成并应用 patch（便于人类本地检查）
- 不会执行 `git commit/push`
- PR 模式下可选发表评论说明（可用 `--no-pr-comments` 关闭）

**3) Patch 应用方式**

使用 `git apply` 应用 patch，而不是直接写文件，降低“写错文件/写坏格式”的风险。

**4) Prompt-based 安全审计（软审计）**

对已修改文件逐个把内容送入模型，要求返回：

```json
{"passed": true, "reason": "..."}
```

当前实现通过 `result.contains("false")` 判定失败，因此它是偏保守的提示机制，不是形式化证明。

### 使用建议

- 在跨仓库复用时，务必提供与目标仓库一致的规则（`--rules-file` 或目标仓库的 `AGENTS.md`）
- 默认不要让自动修复碰 docs/config/lock 文件；必要时显式调整过滤策略
- 在 CI 中开启 `--yes` 前，先用 Dry-Run 观察几轮输出（或在保护分支上只允许 PR 合并）
