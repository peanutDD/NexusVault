- ✅：已落地（或已较完善）
- ⬜：尚未落地 / 仍在规划中

## 二、安全与合规

### 1. 接口安全

**原始规划：**
- 所有写操作鉴权
- 请求限流（per IP / per 用户）
- CSRF：前后端分离 + Bearer Token，避免在跨站 iframe 中暴露接口

**当前状态：**

- ✅ 所有写操作（上传、删除、移动、重命名、分享、组织管理等）统一通过 `AuthenticatedUser` extractor 做鉴权
- ✅ 已实现细粒度限流（自定义中间件）：
  - 全局 IP 级限流：约 60 req/min per IP（可配置窗口与容量）
  - 针对已登录用户的写操作（文件/文件夹/分享/组织）按 `user_id` 做约 120 req/min 的限流
- ⬜ 按端点/套餐等级进一步细化配额与降级策略仍在规划中
- ✅ CSRF 风险整体可控：前后端完全分离 + Bearer Token；OAuth 回调使用 `state` 参数防 CSRF

---

### 2. 存储安全

**原始规划：**
- 文件名 / 路径校验，防目录穿越
- 上传内容校验（MIME / 扩展名白名单）
- 对 HTML / SVG 等危险类型统一 `attachment` / `nosniff`
- 日志脱敏

**当前状态：**

- ✅ 已统一使用安全的文件名清洗与路径拼接逻辑，防止目录穿越（包括 HLS 路径）
- ✅ 通过环境变量 `ALLOWED_MIME_TYPES` / `MAX_FILE_SIZE` 控制上传白名单与最大体积
- ✅ Markdown 图片代理接口具备基础 SSRF 防护（拒绝访问 `localhost/127.0.0.1` 等本地地址）
- ✅ 对 HTML / SVG 等潜在危险类型：
  - 所有下载/预览响应均强制 `Content-Disposition: attachment`
  - 统一添加 `X-Content-Type-Options: nosniff`
  - 即使调用方请求内联预览，也不会在当前页面直接执行
- ✅ 日志对敏感字段（token、cookie、敏感 query 等）做了脱敏和过滤，避免泄露

### 2. 数据库与索引

**原始规划：**
- 为常用查询加索引（`user_id`、`created_at`、文件名搜索等）
- 文件列表分页优化（游标 / keyset）

**当前状态：**

- ✅ 已为高频查询补充复合索引（迁移文件 `016_add_query_performance_indexes.sql`）：
  - `files`：针对用户 + 文件夹 / MIME / 分类 + 创建时间排序的复合索引
  - `files`：针对文件名 / 文件大小排序的复合索引
  - `file_shares`：按文件 + 用户查找、按用户列出分享的索引
  - `folders`：按父文件夹列出子文件夹 + 按名称排序的索引
  - `organization_files`：按组织列出文件的辅助索引
- ✅ 已实现 keyset / 游标分页（cursor-based pagination）：
  - 后端通过 `cursor` 参数实现基于排序字段（`created_at` / `filename` / `file_size`）的游标分页，避免大页码下的 `OFFSET` 性能问题
  - 响应中提供 `next_cursor`，前端可做「加载更多」风格的无限滚动
  - 传统 `page/limit` 分页仍保留，以兼容现有前端与需要跳页的场景

**游标分页的核心价值：**

1. **性能优势（大数据量场景）**：
   - 传统 `OFFSET` 分页：获取第 1000 页需要扫描并跳过前 20000 条记录，查询时间随页码线性增长
   - 游标分页：基于索引直接定位（`WHERE created_at < cursor`），查询时间恒定，不受数据量影响
   - 特别适合无限滚动场景，即使有 10 万+ 文件也能快速加载

2. **数据一致性**：
   - 传统分页：新数据插入会导致已加载页面数据重复或遗漏（如用户在第 1 页时，新文件上传会"挤"到第 1 页，导致第 2 页数据重复）
   - 游标分页：基于时间戳/ID 的稳定锚点，新数据插入不影响已加载页面的数据，适合实时数据流场景

3. **适用场景**：
   - ✅ **最适合**：无限滚动列表、实时数据流（Twitter/Facebook 动态）、大数据量列表（10万+ 记录）
   - ⚠️ **不太适合**：需要跳转到特定页码（如"跳转到第 50 页"）、需要显示总页数/总条数

**原始规划：**

- 构建时开启 brotli/gzip

**当前状态：**

- ✅ 所有主要路由（Login / Register / Files / Settings / Share）均通过 `React.lazy` 懒加载
- ✅ Vite 配置了基于 `manualChunks` 的细粒度代码拆包：

- ✅ 构建阶段已启用 gzip + brotli 压缩（借助 `vite-plugin-compression`）

### 3. 搜索与分类增强（含语义搜索）

**原始规划：**
- 文件名搜索 + 标签 / 分类
- 未来接语义搜索（第一步先把精确搜索做好、做到足够快）

**当前状态：**

- ✅ 已支持按文件名模糊搜索 + 多条件过滤（MIME 类型、时间范围、大小、文件夹等），并配合索引保证性能
- ⬜ 标签 / 分类模型与前端 UI 尚未落地
- ✅ 语义搜索能力已完整实现，并完成与 Hugging Face 的集成：
  - **数据库支持**：
    - 迁移文件 `018_add_semantic_search.sql`：安装 `pgvector` 扩展，在 `files` 表新增 `embedding vector(384)` 列，并建立 IVFFlat + 余弦相似度索引
  - **Hugging Face 集成**：
    - 使用 Hugging Face Inference API 生成 384 维文本嵌入（默认模型 `sentence-transformers/all-MiniLM-L6-v2`）
    - 通过环境变量配置：`HUGGINGFACE_API_TOKEN`（必需）、`HUGGINGFACE_MODEL_ID`（可选）、`HUGGINGFACE_API_URL`（可选）
    - 未配置 Token 时自动禁用语义搜索功能，不影响其他功能
  - **文件内容提取**：
    - 新增 `FileContentExtractor` 服务，支持从多种文件格式中提取文本：
      - 纯文本 / Markdown / JSON / XML / CSV / 日志：直接读取（支持编码检测）
      - PDF：使用 `pdf-extract` crate 提取文本
      - Word（DOCX）：使用 `docx-lite` crate 提取文本（通过临时文件）
      - HTML：去除标签，保留可见文本
    - 生成嵌入时会将「文件名 + 截断后的内容预览（约前 2000 字符）」组合后送入模型
  - **自动向量生成**：
    - 普通上传、分块上传完成、秒传成功后，后台异步任务会自动：
      1. 从存储读取文件内容
      2. 提取文本（使用 `FileContentExtractor`）
      3. 调用 Hugging Face API 生成向量
      4. 写入 `files.embedding` 列
    - 异步执行，不阻塞上传接口响应时间；失败会记录日志，但不影响主流程
  - **语义搜索 API**：
    - 路由：`GET /api/files/search/semantic`
    - 查询参数：
      - `q`（必需）：查询文本
      - `limit`（可选，默认 20）：返回结果数量限制（1-100）
      - `threshold`（可选，默认 0.5）：相似度阈值（0.0-1.0）
    - 响应：返回匹配的文件 ID 列表（按相似度降序）、查询文本、结果数量
    - 未配置 Token 时返回友好错误提示："语义搜索功能未启用，请配置 HUGGINGFACE_API_TOKEN 环境变量"
  - **定价说明**：
    - Hugging Face 免费用户：每月 $0.10 推理额度（约 1000 次请求，取决于模型）
    - Pro 用户：每月 $2.00 额度 + 按需付费（订阅 $9/月）
    - Team/Enterprise：每席位 $2.00 额度
    - 免费额度适合测试和小规模使用，生产环境建议使用 Pro 或自建模型
