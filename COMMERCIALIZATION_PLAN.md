## 二、安全与合规

### 1. 接口安全

- 所有写操作都需要鉴权校验（上传、删除、重命名、移动）
- 请求限流（按 IP / 按用户），防止暴力枚举与滥用
- CSRF：前后端完全分离 + Bearer Token 基本没问题，注意不要在跨站 iframe 中暴露接口

#### 当前状态

- [x] 绝大部分写操作已通过 `AuthenticatedUser` 中间件做鉴权（文件上传 / 删除 / 移动 / 组织操作等）
- [ ] 细粒度的请求限流：已为全站接入基于 IP 的限流（约 60 req/min per IP），并对文件 / 文件夹 / 分享 / 组织等「已登录用户写操作」按 `user_id` 做约 120 req/min 的速率限制；后续仍可按端点与套餐等级进一步细化配额与降级策略
- [x] CSRF 风险整体可控：前后端分离 + Bearer Token；OAuth 回调使用 `state` 做 CSRF 防护

### 2. 存储安全

- 文件名与路径严格校验，防目录穿越（你 HLS 路径已做，其他地方也要统一）
- 上传内容校验：
  - 限制 MIME / 扩展名白名单
  - 对危险类型（如 HTML、SVG 中 JS）增加：
    - `Content-Disposition: attachment`
    - 或 `X-Content-Type-Options: nosniff`
- 日志中避免记录敏感信息：
  - token
  - cookie
  - 完整 URL 中的私密 query

#### 当前状态

- [x] 文件名 / 路径清理与 HLS 路径安全已在后端实现，避免目录穿越
- [x] 已有统一的 MIME 白名单与文件大小限制（`ALLOWED_MIME_TYPES` / `MAX_FILE_SIZE` 环境变量）
- [x] Markdown 图片代理具备 SSRF 防护（禁止访问本地地址）
- [x] 对 HTML / SVG 等潜在危险类型，统一在响应中设置 `X-Content-Type-Options: nosniff`，并在文件/流式响应层强制以 `Content-Disposition: attachment` 返回，即便调用方请求内联预览也不会在当前页面直接渲染执行
- [x] 日志中对路径与错误信息做了脱敏处理，避免泄露敏感字段

### 3. 合规方向（视上线区域决定）

- 提供基本隐私条款 / 使用条款页面
- 定义日志保留期限
- 设计数据删除流程（用户删除账号时真正清理文件和记录）

#### 当前状态

- [ ] 隐私条款 / 使用条款仍需以前端独立页面形式补齐
- [ ] 日志保留期限与数据删除流程目前主要依赖运维实践，尚未形成产品级能力

### 2. 数据库与索引

- 为常用查询加索引（按 `user_id`、`created_at`、文件名搜索等）
- 文件列表分页优化：
  - 游标分页
  - 或 keyset pagination

#### 当前状态

            - [x] 已为高频查询补充复合索引（迁移文件 `016_add_query_performance_indexes.sql`）：
              - files 表：按文件夹/MIME/分类筛选 + 排序的复合索引（user_id + folder_id/mime_type/category + created_at DESC）
              - files 表：按文件名/文件大小排序的复合索引（user_id + original_filename/file_size + ASC/DESC）
              - file_shares 表：按文件+用户查找、按用户列出分享的索引
              - folders 表：按父文件夹列出子文件夹 + 按名称排序的复合索引
              - organization_files 表：按组织列出文件的辅助索引
            - [x] 已实现 keyset / 游标分页（cursor-based pagination）：
              - 后端支持通过 `cursor` 参数使用游标分页（`WHERE sort_column > cursor`），避免 `OFFSET` 在大数据量时的性能问题
              - 游标值根据排序字段类型自动解析（`created_at` 为 ISO 8601 时间戳，`filename` 为字符串，`file_size` 为数字）
              - 响应中包含 `next_cursor` 字段，客户端可使用该值继续获取下一页数据
              - 传统分页（`page` 参数）仍保持向后兼容，默认使用传统分页
              - 前端类型定义已更新以支持游标分页，具体 UI 实现可作为后续优化方向
              
              **游标分页的核心价值：**
              
              1. **性能优势（大数据量场景）**：
                 - 传统 `OFFSET` 分页：获取第 1000 页需要扫描并跳过前 20000 条记录，查询时间随页码线性增长
                 - 游标分页：基于索引直接定位（`WHERE created_at < cursor`），查询时间恒定，不受数据量影响
                 - 特别适合无限滚动场景，即使有 10 万+ 文件也能快速加载
              
              2. **数据一致性**：
                 - 传统分页：新数据插入会导致已加载页面数据重复或遗漏（如用户在第 1 页时，新文件上传会"挤"到第 1 页，导致第 2 页数据重复）
                 - 游标分页：基于时间戳/ID 的稳定锚点，新数据插入不影响已加载页面的数据，适合实时数据流场景
              
              3. **适用场景**：
                 - ✅ **最适合**：无限滚动列表、实时数据流（Twitter/Facebook 动态）、大数据量列表（10万+ 记录）
                 - ⚠️ **不太适合**：需要跳转到特定页码（如"跳转到第 50 页"）、需要显示总页数/总条数
              
              4. **使用建议**：
                 - 文件列表默认使用游标分页（不传 `page`，传 `cursor`），提供更好的无限滚动体验
                 - 保留传统分页作为备选方案（需要跳转页码时使用）
                 - 前端实现"加载更多"按钮，而不是页码导航

### 3. 前端性能

- 按路由拆包、懒加载非核心组件（如高级预览、设置页）
- 构建阶段开启 brotli / gzip 压缩
- 配置合理的 `Cache-Control`

#### 当前状态

- [x] 已在文件列表等场景引入虚拟列表、请求去重与本地缓存，整体性能较好
- [x] **路由懒加载**：所有路由（Login、Register、Files、Settings、Share）均已使用 `React.lazy()` 实现按需加载
- [x] **代码拆包**：已配置 `manualChunks` 将 vendor 库分离（vendor-react、vendor-form、vendor-state、vendor-utils），提升缓存命中率
- [x] **资源压缩**：已配置 `vite-plugin-compression`，构建时自动生成 gzip（`.gz`）和 brotli（`.br`）压缩文件
- [x] **静态资源缓存**：Nginx 配置了静态资源（JS/CSS/图片/字体）1 年缓存 + `Cache-Control: public, immutable`
- [x] **PWA 运行时缓存**：配置了 Workbox 缓存策略（文件列表/文件夹 API 5 分钟缓存，预览文件 7 天缓存）
- [ ] **可进一步优化**：
  - 按路由更细粒度拆分代码（当前按 vendor 拆分，可考虑按路由拆分如 `chunk-Files`、`chunk-Settings`）
  - HTML 文件缓存策略优化（当前 Nginx 未配置 HTML 的 Cache-Control，建议设置为 `no-cache` 或 `max-age=0`）
  - 添加 `build:analyze` 脚本用于分析 bundle 大小（已配置 visualizer，但需确认 package.json 中是否有对应脚本）

## 八、优先级建议：先做哪几件事

按「性价比」排序，推荐你优先做：

1. **账号与权限体系打通**

2. **安全加固**
   - 上传白名单
   - 危险文件头处理
   - 限流
3. **监控与日志**
   - 接入 Sentry
   - 梳理和强化 tracing（你现在已经在用，继续整理 `request_id`）
4. **部署一套稳定的 prod 环境**
   - 独立数据库
   - 对象存储
   - 反向代理（Nginx / Caddy）
   - HTTPS
