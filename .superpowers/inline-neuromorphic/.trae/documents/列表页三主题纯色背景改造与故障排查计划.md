# 列表页三主题纯色背景改造与故障排查计划

## 目标与验收标准

- 将列表页在 `light`、`dark`、`purple` 三个主题下的背景从“任何渐变形式”统一改为“纯色”。
- 改造范围覆盖页面主背景、列表容器背景、筛选区域背景、工具栏与强调态背景，避免局部残留渐变。
- 验收以“代码定义 + 运行态视觉 + 样式来源追踪”三层一致为准：主题切换后不出现线性/径向渐变。

## 现状链路与风险点

### 主题链路

1. `themeStore` 负责将主题写入 `data-theme` 与主题 class。
2. `tokens.css` 在 `:root` 与各主题块中定义并覆写背景相关变量。
3. 列表页与布局组件通过 CSS 变量消费背景定义。

### 当前渐变来源（需全部清除）

1. 页面级变量 `--surface-page-gradient*` 被 `PageLayout` 的背景图片类引用。
2. 文件列表相关变量（`--filelist-*bg*`）在基础与主题块存在渐变定义。
3. 筛选区相关变量（`--filters-surface-*`）存在渐变或混合背景。
4. 局部选择器存在硬编码渐变（按钮高亮、选中项等），绕过 token 直接渲染。

## 实施步骤

### 第一步：建立“背景项全量清单”

1. 以列表页实际渲染路径为主线，枚举所有会影响背景视觉的变量与选择器。
2. 将背景项按四类归档：页面主背景、列表面板、筛选面板、交互高亮。
3. 为每一项标记来源文件、选择器/变量名、是否为硬编码渐变。

交付物：
- 一份可执行改造清单，保证不存在遗漏项。

### 第二步：统一主题 token 为纯色

1. 在 `tokens.css` 中将 `light/dark/purple` 对应的页面与列表背景变量改为纯色值。
2. 对基础主题块同步校正，避免主题未命中时回退到渐变。
3. 保持变量语义不变，只替换取值，减少组件层改动面。

验收点：
- 所有目标变量值为纯色（如 `#xxxxxx` 或 `rgb/oklch` 单色），不包含 `linear-gradient/radial-gradient/conic-gradient`。

### 第三步：清理组件级硬编码渐变

1. 在列表与筛选 CSS 中定位并替换硬编码渐变选择器。
2. 优先改为引用统一 token；若无合适 token，补充新 token 并在三主题统一赋纯色。
3. 检查 hover/active/selected/强调按钮态，避免交互态恢复为渐变。

验收点：
- 列表页相关 CSS 中不再保留背景渐变硬编码。

### 第四步：校正布局背景引用方式

1. 审核 `PageLayout` 对 `--surface-page-gradient` 的使用方式。
2. 若语义与实现不匹配（变量名虽为 gradient 但需纯色），统一为“纯色可稳定消费”的方式：
   - 要么保留变量名但值改纯色并确保渲染正确；
   - 要么改为背景色变量引用并同步替换调用处。
3. 保证三主题切换后页面底色一致受控。

验收点：
- 页面根容器背景渲染为纯色，且主题切换即时生效。

### 第五步：彻底验证与故障排查闭环

1. 静态验证：检索列表页相关样式与 token，确认无渐变关键字残留。
2. 运行验证：在 `light/dark/purple` 三主题分别进入列表页，核对页面主背景、列表区、筛选区、选中态是否纯色。
3. 若“看起来仍像渐变”，按以下顺序排查并修正：
   - 样式优先级与覆盖顺序（是否被后定义规则覆盖）；
   - 主题命中条件（`data-theme` 与 class 是否一致）；
   - 变量回退链（`var()` fallback 是否指向渐变值）；
   - 构建缓存/热更新缓存导致旧样式残留；
   - 非背景属性造成视觉假象（阴影/透明叠层/滤镜）。
4. 每发现一个根因，立即修正并回归三主题复测，直到全部通过。

验收点：
- 三主题下列表页全域背景均为纯色，且不存在“局部、状态态、回退态”渐变。

## 变更边界与约束

- 仅调整与“列表页背景纯色化”直接相关的样式与主题变量，不引入无关视觉改版。
- 不改变主题切换机制与业务逻辑。
- 保持现有组件结构与命名风格，优先最小侵入改造。

## 完成判定

满足以下条件即判定完成：

1. 代码层：目标文件不存在列表页背景相关渐变定义。
2. 运行层：`light/dark/purple` 三主题下列表页可视背景均为纯色。
3. 排查层：若过程中出现未生效问题，已定位根因并完成修正，最终回归通过。
