# 原生 Android 新项目建设与 UI 升级实施计划（Plan）

## 1. 目标与边界

* 在仓库内新增一个**原生 Android 客户端项目**，与现有 `frontend`（Web/Tauri）并行共存，旧项目保留且不删除。

* 原生端以 Kotlin + Jetpack Compose 为核心，覆盖当前核心业务链路（登录、文件列表、上传下载、预览、设置）。

* 同步进行 UI/UX 体系升级，形成统一设计语言与高可用交互规范，提升移动端体验。

* 本计划聚焦“新建项目 + 架构设计 + 技术选型 + UI 设计方案”，执行阶段按增量迁移落地。

***

## 2. 总体方案：双轨并存，不替换旧项目

### 2.1 仓库级组织策略

* 保留现有 `frontend`、`frontend/src-tauri` 作为旧端（Web/Tauri）。

* 在仓库根目录新增独立目录（建议）：

  * `android-native/`：原生 Android 全量工程。

* 新旧端共享后端 API，不共享 UI 代码；通过接口契约与设计规范实现一致体验。

### 2.2 迁移策略

* **阶段性并行**：先实现原生 MVP，旧端继续服务。

* **按功能分批迁移**：认证 → 文件列表/筛选 → 上传下载 → 预览/分享 → 设置。

* **灰度验收**：每一阶段都通过可安装包和真实设备冒烟验证，再推进下一阶段。

***

## 3. 原生项目架构设计（重点）

### 3.1 架构原则

* 采用 Clean Architecture + MVVM + UDF（单向数据流）。

* 强约束分层：UI 层不直接依赖网络/数据库实现。

* 默认离线友好、弱网可用、状态可恢复。

### 3.2 模块拆分（建议）

* `:app`\
  应用入口、导航容器、Hilt 装配、全局配置。

* `:core:designsystem`\
  主题、色板、字体、组件库、动效规范。

* `:core:ui`\
  通用 UI 容器（状态页、列表骨架、错误态、加载态、对话框等）。

* `:core:network`\
  Retrofit/OkHttp/Kotlinx Serialization、鉴权拦截器、错误映射。

* `:core:database`\
  Room、本地缓存、DAO、迁移策略。

* `:core:data`\
  Repository 实现、Remote+Local 协同、缓存策略。

* `:core:domain`\
  UseCase、领域模型、仓库接口（纯 Kotlin）。

* `:core:common`\
  常量、Result 封装、调度器、日志接口。

* `:feature:auth`

* `:feature:files`

* `:feature:upload`

* `:feature:preview`

* `:feature:settings`

### 3.3 数据流规范

* UI 通过 ViewModel 暴露 `StateFlow<UiState>`，事件以 `UiEvent` 上送。

* 业务通过 UseCase 编排，Repository 统一聚合多数据源。

* 网络错误统一映射为领域错误模型，保证界面可预测呈现。

***

## 4. 技术选型（主流且可长期演进）

### 4.1 必选技术栈

* 语言：Kotlin（最新稳定版）。

* UI：Jetpack Compose + Material 3。

* 架构：MVVM + Clean Architecture。

* DI：Hilt。

* 并发：Kotlin Coroutines + Flow。

* 导航：Navigation Compose（类型安全路由方案）。

* 网络：Retrofit + OkHttp + Kotlinx Serialization。

* 本地存储：Room（结构化）+ DataStore（配置/轻量状态）。

* 图片：Coil。

* 分页：Paging 3（大列表性能关键）。

* 后台任务：WorkManager（上传重试、后台同步）。

* 日志/监控：Timber + Crashlytics（或等价方案）。

* 测试：JUnit5、MockK、Turbine、Compose UI Test、Macrobenchmark。

### 4.2 工程与质量

* 构建：Gradle Kotlin DSL + Version Catalog。

* 代码质量：ktlint + detekt + Android Lint。

* CI：PR 触发单元测试、静态检查、UI smoke（可选设备云）。

* 发布：分 flavor（dev/stage/prod）+ 签名与版本策略标准化。

***

## 5. 哪些内容必须设计为原生结构

* 认证与会话管理：令牌续期、设备级安全存储（Encrypted Storage）。

* 文件列表与筛选：原生分页、下拉刷新、骨架屏与空态。

* 上传下载链路：断点续传、后台重试、系统通知进度、失败可恢复。

* 预览系统：图片/视频/PDF/文本采用原生能力分发。

* 本地缓存与离线浏览：最近访问、元数据缓存、网络恢复自动同步。

* 系统级交互：分享面板、文件选择器、权限请求、前后台生命周期处理。

***

## 6. 可优化方向（性能 + 体验）

* 启动优化：Splash 路由最短路径、延迟初始化非关键模块、Baseline Profile。

* 列表性能：Paging + 稳定 key + 懒加载占位 + 图片解码策略。

* 网络体验：请求去重、重试退避、弱网降级、超时与取消机制。

* 操作反馈：全链路加载/成功/失败反馈统一，避免“无响应感”。

* 可访问性：动态字号、屏幕阅读支持、对比度与触控热区规范。

* 国际化与主题：深浅色 + 品牌主题 + 文案体系统一。

***

## 7. UI 设计升级方案（资深设计师视角）

### 7.1 设计系统（Design System）

* 建立 Token 体系：颜色、间距、圆角、阴影、字号、层级、动效时长。

* 组件分层：基础组件（按钮/输入框）→ 业务组件（文件卡片/筛选条）→ 页面模板。

* 明确状态矩阵：默认、悬浮、按下、禁用、错误、加载、空数据。

### 7.2 视觉风格建议

* 采用“高信息密度 + 清晰层级”的生产力风格，而非装饰性过强视觉。

* 主页强化主任务入口（上传、搜索、最近文件、快捷筛选）。

* 颜色上以中性灰阶 + 品牌强调色，确保长期可读与深色模式一致性。

### 7.3 关键交互改进

* 文件操作采用“底部动作面板 + 手势补充”，单手操作优先。

* 上传流程做“任务中心化”：可暂停、可重试、可查看失败原因。

* 预览页强化“沉浸式浏览 + 快捷操作”（下载、分享、删除、重命名）。

* 设置页采用分组卡片，优先展示账户、存储、主题、安全项。

***

## 8. 实施阶段计划（执行时严格按序）

1. 新建 `android-native` 工程与多模块骨架，接入 Hilt、Compose、导航框架。
2. 搭建 `core` 层（network/database/data/domain/common），完成基础能力。
3. 完成 `feature:auth`（登录注册、会话管理、错误态）。
4. 完成 `feature:files`（分页列表、筛选、文件夹层级、空态/骨架）。
5. 完成 `feature:upload` 与后台任务（重试、通知、失败恢复）。
6. 完成 `feature:preview` 与系统分享链路。
7. 完成 `feature:settings`（主题、账号、存储、API Token 等）。
8. 落地设计系统与全局 UI 重构，统一动效与反馈规范。
9. 建立测试与性能基线，执行真机回归、弱网回归、稳定性验证。
10. 输出迁移报告与并行运行策略文档，确保旧端与新端长期共存可维护。

***

## 9. 交付物定义

* 新增 `android-native/` 原生工程（多模块可编译）。

* 技术架构文档（模块图、数据流、错误模型、缓存策略）。

* UI 规范文档（Design Tokens、组件规范、页面模板、交互规则）。

* 核心流程可运行 APK（dev/stage），含基础监控与日志。

* 验收报告（功能覆盖、性能指标、体验评估、已知风险）。

***

## 10. 验收标准（DoD）

* 旧端保留且可正常构建运行；新端可独立构建安装。

* 新端核心链路全部可用：登录、列表、上传下载、预览、设置。

* UI 风格统一、状态反馈完整、弱网场景可恢复。

* 性能达标：冷启动、列表滚动、上传稳定性达到预设阈值。

* 文档完整，团队可按文档继续迭代并保持双端并行维护。
