## 🔀 请求链路简图（文件列表）

1. **用户操作**（搜索/筛选/翻页/进文件夹）→ 2. **防抖/节流**（搜索 300ms；loadMore 400ms）→ 3. **请求取消**（同 key 的上一请求被 abort）→ 4. **全局限流**（`limitedApi` 排队，最多 20 并发）→ 5. **请求去重**（adapter 层：同 key 5s 内复用缓存或飞行中 Promise）→ 6. **列表缓存 + SWR**（有 localStorage 缓存则先展示并设 isRevalidating，后台请求完成后更新并写回缓存；无缓存则直接请求）→ 7. **loadMore**（若该页有缓存则直接用缓存不请求）→ 8. **缓存失效**（移动/批量移动/登出后调用 `clearFileListCache()`）。

## 一、请求频率控制（最直接有效）

### 1.1 防抖（Debounce）✅ 已实现

**原理**：延迟执行，只在用户停止操作后触发请求。

**当前实现**：

- ✅ 搜索输入：300ms 防抖（`useFileList` 内 `useDebounce(search, 300)` → `debouncedSearch` 参与 `getCacheKey` 与请求）
- ✅ 位置：`frontend/src/hooks/useDebounce.ts`；使用处：`useFileList.ts`

**优化建议**：可针对不同场景调整延迟（如筛选器 500ms、排序 200ms），当前仅搜索防抖以平衡体验与请求量。

### 1.2 节流（Throttle）✅ 已实现

**原理**：限制执行频率，固定时间间隔内最多执行一次。

**当前实现**：

- ✅ 值节流：`frontend/src/hooks/useThrottle.ts`（对值做节流更新）。
- ✅ 回调节流：`frontend/src/hooks/useThrottledCallback.ts`（对函数调用做 leading 节流）。
- ✅ 已接入场景：
  - **无限滚动**：`FileList.tsx` 中 `loadMore` 经 `useThrottledCallback(loadMore, 400)` 后传给 `InfiniteScrollSentinel` 与 `FileListPagination`，避免触底时重复请求。
  - **预览切换**：`FilePreview.tsx` 中 `goToPrev` / `goToNext` 使用 `useThrottledCallback(..., 200)`，限制快速点击导致的预览请求频率。
  - **resize**：`VirtualizedFileGrid.tsx` 列数更新与视口更新、`FileListFilters.tsx` 下拉菜单位置更新，resize 监听均做了 150ms 节流，减少布局计算与重绘。

**预期效果**：接入后滚动/预览/resize 相关请求与计算 ↓ 70-90%

## 二、请求去重与合并

### 2.1 请求去重（Dedup）✅ 已实现（全局 + TTL）

**原理**：相同参数的请求在飞行中只发一次，后续请求复用同一个 Promise；完成后在 TTL 内相同请求直接返回缓存结果，不重复发请求。

**当前实现**：

- ✅ **全局 adapter 包装**：`frontend/src/utils/globalRequestDedup.ts` 的 `createDedupAdapter` 包装 axios 默认 adapter，对所有通过 `api` 发出的请求做去重与 TTL 缓存；`api.ts` 初始化时设置 `api.defaults.adapter = createDedupAdapter(adapterFn)`。
- ✅ **TTL**：`REQUEST.DEDUP_TTL_MS`（5s）内相同 key 的请求：若已有缓存则直接返回缓存响应；若请求在飞行中则复用同一 Promise。
- ✅ **缓存条件**：仅缓存 2xx 且 `responseType` 非 `blob`/`arraybuffer`/`stream`（避免误缓存下载/预览二进制）。
- ✅ **Key**：`getDedupKey(config)` = `method` + `fullUrl`（含 params）+ `dataPart`（JSON 或字符串；FormData/Blob 不参与，避免上传/下载被误合并）。
- ✅ **缓存上限**：`REQUEST.DEDUP_MAX_CACHE_SIZE`（100），超限时先清理过期条目再按插入顺序淘汰。
- ✅ 业务层可选 `useRequestDedup`（`frontend/src/hooks/useRequestDedup.ts`）做组件级去重，与全局去重叠加生效。

### 2.2 批量请求合并 ✅ 已实现并接入

**原理**：将多个独立请求合并成一个批量接口调用。

**当前实现**：

- ✅ **后端**：`POST /api/files/batch`，请求体 `{ ids: string[] }`，返回 `{ files: (FileMetadata | null)[] }`（顺序与 ids 一致，未找到为 null）。单次最多 100 个 ID。
- ✅ **前端**：`frontend/src/utils/batchRequest.ts` 的 `BatchRequestManager`；`fileService.getFilesByIds(ids)` 直接调用批量接口；`fileService.getFileMetadata(id)` 经 `fileMetadataBatch.request('metadata', id)` 在 `REQUEST.BATCH_DELAY_MS`（50ms）窗口内同 key 合并为一次批量请求。
- ✅ **业务接入**：`BatchShareDialog` 打开时调用 `getFilesByIds(fileIds)` 拉取文件名并展示「将创建分享：N 个文件（…）」。其他按 id 查详情的场景可改用 `getFileMetadata(id)` 以自动合并。

**预期效果**：接入后批量操作/按 id 查详情请求数 ↓ 80-95%

## 三、请求取消与竞态处理

### 3.1 请求取消 ✅ 已实现

**原理**：后发的请求取消前发的，避免无效响应覆盖最新数据。

**当前实现**：

- ✅ `api.ts`：`shouldCancelRequest(url)` 仅对 `url.includes('/api/files') && url.includes('?')` 返回 true；`getRequestKey(config)` = `method` + `fullUrl`（含 params）。请求拦截中为符合条件的 GET 创建/复用 AbortController 并设置 `config.signal`；响应拦截（成功或失败）中调用 `clearAbortController(config)` 从 Map 中移除该 key，避免误取消文件夹/预览等。

  - **后端**（`backend/src/services/file/batch_zip.rs` + `handlers/files/batch.rs`）：POST /download-zip 流式响应。`run_zip_writer_thread` 使用 `ChannelWriter` 边打包边通过 channel 发送；`ChannelWriter` 缓冲 8KB 即发、**每写完一个文件调用 flush**，首包更早、小文件也能立刻发出；响应体不整包进内存。
  - **前端**（`frontend/src/services/files.ts` `downloadZip`）：在支持 **File System Access API** 的浏览器（Chrome、Edge 等）下，先调用 `showSaveFilePicker({ suggestedName: 'files.zip' })`，**保存对话框立即出现**；用户选路径后，用 `fetch` 流式请求 ZIP 并 `response.body` 通过 `handle.createWritable()` 流式写入。**成功**时在 `finally` 中调用 `writable.close()`，Chrome 才会把临时文件替换成 `files.zip`；**失败**时调用 `writable.abort()` 丢弃不完整文件，避免留下半截内容。不支持 File System Access API 时（如 Firefox、Safari）降级为 axios blob 下载。
  - `**.crswap` 说明**：Chrome 流式保存时会先写到临时文件（名称常带 `.crswap`），只有成功调用 `close()` 后才会替换成目标 `files.zip`。若只看到 `files.zip.crswap` 或打不开，多为写入未完成或未正确 close；可尝试将 `files.zip.crswap` 重命名为 `files.zip` 看是否完整（若为中断则仍可能损坏）。当前实现已保证成功路径必 `close()`、失败路径 `abort()`。
- ✅ **数据库连接池**（高并发/批量下载时减少「pool timed out」）：后端 `backend/src/database/pool.rs` 支持环境变量 `DB_POOL_MAX_CONNECTIONS`（默认 40）、`DB_POOL_ACQUIRE_TIMEOUT_SECS`（默认 15）；见 `backend/.env.example` 注释。

**预期效果**：全选即选全；多文件夹/多同名文件批量下载 ZIP 不再报错；支持时保存框立即出现、边下边写；首包更早、大包更稳；高并发下连接池不易超时。

## 📝 注意事项

1. **缓存失效策略**：移动文件/文件夹、批量移动、登出后已调用 `clearFileListCache()`（见 4.3）；删除文件、上传完成后当前依赖 SWR 后台 revalidate 或用户切换条件后新请求，若需「删除/上传后立即让其它页缓存失效」可在对应成功回调中补充 `clearFileListCache()`。Service Worker 运行时缓存与 localStorage 列表缓存独立，SW 按 URL 缓存 GET 响应，过期由 Workbox 的 `maxAgeSeconds`/`maxEntries` 控制。
2. **错误处理**：所有优化手段需有降级方案（如缓存失败不影响主流程、revalidate 失败静默不覆盖已有展示、重试失败后不再无限重试）。
3. **监控**：可添加请求数、去重命中率、列表缓存命中率、SW 缓存命中率、错误率等监控，便于评估与调优。
4. **测试**：高并发与弱网场景下验证请求取消、限流、去重、SWR、SW 离线回退等行为符合预期。
5. **全选与批量 ZIP**：全选包含当前列表所有文件与文件夹；批量下载 ZIP 时后端对重名文件使用「base (1).ext」等唯一条目名，避免 Duplicate filename。
6. **批量下载 ZIP**：后端 POST /download-zip **流式**（`batch_zip.rs`：ChannelWriter 8KB 即发、每文件后 flush）；前端在支持时用 **File System Access API** 先弹保存框再流式写入（成功 `writable.close()`、失败 `writable.abort()`，避免 `.crswap` 残留），不支持时降级 blob。点击「Batch Download ZIP」后按钮为「打包中…」并禁用。连接池见 `DB_POOL_MAX_CONNECTIONS` / `DB_POOL_ACQUIRE_TIMEOUT_SECS`（默认 40/15s）。

## ✅ 方案完整性检查清单

核对时可按下列项逐项确认（均以 `frontend/src` 为准）：

- **防抖**：搜索使用 `useDebounce(search, 300)`，`debouncedSearch` 参与列表请求
- **节流**：`loadMore` 经 `useThrottledCallback(..., 400)`；预览切换 200ms；resize 150ms（VirtualizedFileGrid、FileListFilters）
- **全局去重**：`api.defaults.adapter` 为 `createDedupAdapter(...)`，TTL 5s、100 条上限，仅缓存 2xx 且非 blob/arraybuffer
- **批量合并**：`fileService.getFileMetadata` 经 `fileMetadataBatch`，50ms 窗口；`getFilesByIds` 直调 `POST /api/files/batch`；BatchShareDialog 使用 `getFilesByIds`
- **请求取消**：仅 `GET /api/files?` 同 key 取消前一个，响应后清理 AbortController
- **全局限流**：`listFiles` 使用 `limitedApi.get`，LIMITER_MAX_CONCURRENT=20
- **列表缓存**：`fileListCache` 版本号 + LRU 80 条，5min TTL；loadMore 有缓存则直接追加不请求
- **SWR**：`loadFiles` 有缓存则先展示再后台 revalidate，用 `fileListCacheKeyRef` 防覆盖，暴露 `isRevalidating`
- **上传队列**：`UploadQueue` 使用 MAX_COST=3、LARGE_FILE_THRESHOLD_BYTES=10MB；`UploadDialog` 传 `fileSize`、`priority: totalPending - index`
- **乐观更新-重命名**：`RenameFolderDialog` 使用 `onRename={handleRenameFolderSubmit}`，先 setFolders 再请求，失败回滚
- **乐观更新-移动**：`handleDropOnFolder`、`handleDropOnBreadcrumb`、`BatchMoveDialog` 使用 `getOptimisticMoveRollback`，目标≠当前目录时先移除再请求，失败回滚
- **缓存失效**：移动/批量移动/登出后调用 `clearFileListCache()`
- **虚拟列表**：`files.length > FILE_LIST.VIRTUAL_THRESHOLD` 时使用 VirtualizedFileGrid
- **无限滚动**：InfiniteScrollSentinel 触发 loadMore，FileListPagination 支持「加载更多」模式
- **Service Worker**：生产环境注册，静态预缓存 + 文件列表/文件夹 API NetworkFirst
- **React Compiler**：vite.config 中 react babel 插件含 babel-plugin-react-compiler
- **startTransition**：loadFiles/loadMore 中列表更新（setFiles/setTotal 等）包在 startTransition
- **content-visibility**：VirtualizedFileGrid 行使用 .virtualized-row（content-visibility: auto）
- **Web Worker 分组**：文件数>50 且按类型分组时使用 groupFiles.worker，groupedFiles 用 worker 结果或 memoGrouped
- **全选包含文件+文件夹**：useSelectionState(files, folders)，全选时选中所有文件与文件夹
