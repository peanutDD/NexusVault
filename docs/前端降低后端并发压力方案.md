## 🔀 请求链路简图（文件列表）

1. **用户操作**（搜索/筛选/翻页/进文件夹）→ 2. **防抖/节流**（搜索 300ms；loadMore 400ms）→ 3. **请求取消**（同 key 的上一请求被 abort）→ 4. **全局限流**（`limitedApi` 排队，最多 20 并发）→ 5. **请求去重**（adapter 层：同 key 5s 内复用缓存或飞行中 Promise）→ 6. **列表缓存 + SWR**（有 localStorage 缓存则先展示并设 isRevalidating，后台请求完成后更新并写回缓存；无缓存则直接请求）→ 7. **loadMore**（若该页有缓存则直接用缓存不请求）→ 8. **缓存失效**（移动/批量移动/登出后调用 `clearFileListCache()`）。

## 一、请求频率控制（最直接有效）

### 1.1 防抖（Debounce）✅ 已实现

**原理**：延迟执行，只在用户停止操作后触发请求。

**当前实现**：
- ✅ 搜索输入：300ms 防抖（`useFileList` 内 `useDebounce(search, 300)` → `debouncedSearch` 参与 `getCacheKey` 与请求）
- ✅ 位置：`frontend/src/hooks/useDebounce.ts`；使用处：`useFileList.ts`

**优化建议**：可针对不同场景调整延迟（如筛选器 500ms、排序 200ms），当前仅搜索防抖以平衡体验与请求量。

### 1.2 节流（Throttle）✅ 已实现

**原理**：限制执行频率，固定时间间隔内最多执行一次。

**当前实现**：
- ✅ 值节流：`frontend/src/hooks/useThrottle.ts`（对值做节流更新）。
- ✅ 回调节流：`frontend/src/hooks/useThrottledCallback.ts`（对函数调用做 leading 节流）。
- ✅ 已接入场景：
  - **无限滚动**：`FileList.tsx` 中 `loadMore` 经 `useThrottledCallback(loadMore, 400)` 后传给 `InfiniteScrollSentinel` 与 `FileListPagination`，避免触底时重复请求。
  - **预览切换**：`FilePreview.tsx` 中 `goToPrev` / `goToNext` 使用 `useThrottledCallback(..., 200)`，限制快速点击导致的预览请求频率。
  - **resize**：`VirtualizedFileGrid.tsx` 列数更新与视口更新、`FileListFilters.tsx` 下拉菜单位置更新，resize 监听均做了 150ms 节流，减少布局计算与重绘。

**预期效果**：接入后滚动/预览/resize 相关请求与计算 ↓ 70-90%

## 二、请求去重与合并

### 2.1 请求去重（Dedup）✅ 已实现（全局 + TTL）

**原理**：相同参数的请求在飞行中只发一次，后续请求复用同一个 Promise；完成后在 TTL 内相同请求直接返回缓存结果，不重复发请求。

**当前实现**：
- ✅ **全局 adapter 包装**：`frontend/src/utils/globalRequestDedup.ts` 的 `createDedupAdapter` 包装 axios 默认 adapter，对所有通过 `api` 发出的请求做去重与 TTL 缓存；`api.ts` 初始化时设置 `api.defaults.adapter = createDedupAdapter(adapterFn)`。
- ✅ **TTL**：`REQUEST.DEDUP_TTL_MS`（5s）内相同 key 的请求：若已有缓存则直接返回缓存响应；若请求在飞行中则复用同一 Promise。
- ✅ **缓存条件**：仅缓存 2xx 且 `responseType` 非 `blob`/`arraybuffer`/`stream`（避免误缓存下载/预览二进制）。
- ✅ **Key**：`getDedupKey(config)` = `method` + `fullUrl`（含 params）+ `dataPart`（JSON 或字符串；FormData/Blob 不参与，避免上传/下载被误合并）。
- ✅ **缓存上限**：`REQUEST.DEDUP_MAX_CACHE_SIZE`（100），超限时先清理过期条目再按插入顺序淘汰。
- ✅ 业务层可选 `useRequestDedup`（`frontend/src/hooks/useRequestDedup.ts`）做组件级去重，与全局去重叠加生效。

### 2.2 批量请求合并 ✅ 已实现并接入

**原理**：将多个独立请求合并成一个批量接口调用。

**当前实现**：
- ✅ **后端**：`POST /api/files/batch`，请求体 `{ ids: string[] }`，返回 `{ files: (FileMetadata | null)[] }`（顺序与 ids 一致，未找到为 null）。单次最多 100 个 ID。
- ✅ **前端**：`frontend/src/utils/batchRequest.ts` 的 `BatchRequestManager`；`fileService.getFilesByIds(ids)` 直接调用批量接口；`fileService.getFileMetadata(id)` 经 `fileMetadataBatch.request('metadata', id)` 在 `REQUEST.BATCH_DELAY_MS`（50ms）窗口内同 key 合并为一次批量请求。
- ✅ **业务接入**：`BatchShareDialog` 打开时调用 `getFilesByIds(fileIds)` 拉取文件名并展示「将创建分享：N 个文件（…）」。其他按 id 查详情的场景可改用 `getFileMetadata(id)` 以自动合并。

**预期效果**：接入后批量操作/按 id 查详情请求数 ↓ 80-95%

### 3.1 请求取消 ✅ 已实现

**原理**：后发的请求取消前发的，避免无效响应覆盖最新数据。

**预期效果**：无效列表/搜索请求 ↓ 40-70%

## 四、缓存策略

### 4.1 本地缓存 ✅ 已实现

**当前实现**：
- ✅ 文件列表缓存（localStorage）：`frontend/src/utils/fileListCache.ts`；`getCacheKey(query)` = `file_list_cache_` + 排序后的 `key=value&...`。
- ✅ **TTL**：`CACHE_DURATION = FILE_LIST.CACHE_MINUTES * 60 * 1000`（5 分钟）；读取时若 `now - data.timestamp > CACHE_DURATION` 则视为过期并删除。
- ✅ **缓存版本号**：`FILE_LIST.CACHE_VERSION`（1）；每条缓存带 `version`，读取时若与当前版本不一致则删除并返回 null，数据结构变更时递增版本即可使旧缓存失效。
- ✅ **LRU 淘汰**：`FILE_LIST.CACHE_MAX_ENTRIES`（80）；LRU 顺序存于 `file_list_cache::lru`（JSON 数组）；读命中时 `touchLru(key)` 将该 key 移到末尾；写时 `evictLru()` 若超过最大条数则从头部逐条删除；写入失败时先执行一次 LRU 淘汰再重试。
- ✅ **loadMore 使用缓存**：`useFileList` 的 `loadMore` 中若该页 `getCachedFileList(cacheKey)` 有值，则直接追加到列表并 `setLoadedPageCount(nextPage)`，不发请求。

### 4.2 stale-while-revalidate ✅ 已实现并接入（文件列表）

**原理**：先用缓存显示，后台静默更新，更新后替换。

**当前实现**：
- ✅ `frontend/src/hooks/useStaleWhileRevalidate.ts`：内存缓存 + TTL，返回 `{ data, isValidating, mutate }`（供其他场景复用）。
- ✅ **文件列表已接入**：`useFileList` 的 `loadFiles`：有 localStorage 缓存时先 `setFiles(cached.files)`、`setTotal(cached.total)`、`setLoading(false)`、`setIsRevalidating(true)`，然后后台 `dedupedListFiles(query)`，完成后若 `fileListCacheKeyRef.current === cacheKey` 则更新 `files`/`total` 并 `setCachedFileList`，最后 `setIsRevalidating(false)`；无缓存时直接请求并写回缓存。用 `fileListCacheKeyRef` 避免筛选变更后旧 revalidate 覆盖新结果；暴露 `isRevalidating`，列表栏在后台更新时显示「更新中…」。

**预期效果**：首屏/切回已访问条件时先出缓存，感知延迟降低；后台更新后数据保持新鲜。

### 4.3 缓存失效 ✅ 已接入

**当前实现**：
- ✅ `clearFileListCache()`（`fileListCache.ts`）：清空所有 `file_list_cache_*` 及 `file_list_cache::lru`。
- ✅ **调用场景**：登出时（`Files.tsx`、`FileList.tsx`、`useFileList.ts` 内登出逻辑）；移动文件到文件夹（`handleDropOnFolder` 成功后）；移动文件夹（`handleDropOnFolder` 成功后）；拖拽到面包屑（`handleDropOnBreadcrumb` 成功后）；批量移动（`BatchMoveDialog` 应用成功后，`useFileList` 内 `clearFileListCache()`）。
- ⚠️ 删除文件、上传完成等会触发列表重新加载（`loadFiles`/`loadFolders`），但未显式调用 `clearFileListCache`，依赖 SWR 后台 revalidate 或用户切换条件后新请求；若需「删除/上传后立即让其它页缓存失效」，可在对应成功回调中补充 `clearFileListCache()`。

### 5.1 上传队列 ✅ 已实现（按成本并发 + 优先级）

**当前实现**：
- ✅ `frontend/src/utils/uploadQueue.ts`：`UploadQueue` 类，构造函数 `(maxCost = UPLOAD_QUEUE.MAX_COST, largeThresholdBytes = UPLOAD_QUEUE.LARGE_FILE_THRESHOLD_BYTES)`。

**优化建议**：可根据业务需要调整 `MAX_COST` 或 `LARGE_FILE_THRESHOLD_BYTES`；优先级已支持，可按「重要文件优先」传入更大 priority。

## 六、乐观更新 ✅ 已实现并部分接入

**原理**：先更新 UI，再发请求，失败时回滚。

**当前实现**：
- ✅ **移动（拖拽/批量移动）**：`useFileList` 内 `getOptimisticMoveRollback(fileIds, folderIds, targetFolderId)`：当目标非当前目录时，先从 `files`/`folders` 中移除被移动项，返回回滚函数；请求失败时调用回滚并恢复原列表。用于 `handleDropOnFolder`（拖到文件夹）、`handleDropOnBreadcrumb`（拖到面包屑）、`BatchMoveDialog` 的 `onApplyOptimistic`（批量移动前乐观移除，成功后清缓存，失败回滚）。
- ✅ **重命名文件夹**：`handleRenameFolderSubmit`：先 `setFolders(prev => prev.map(...))` 更新名称，再请求 `folderService.rename`，失败时用保存的 `prevFolders` 回滚并 rethrow。
- ✅ **通用 Hook**：`frontend/src/hooks/useOptimisticUpdate.ts` 提供 `useOptimisticUpdate(initial, updateFn)`，返回 `[state, update]`，`update(optimistic, actualPromise)` 先设乐观值、成功更新、失败回滚；当前未在其他业务中接入，可按需在其它写操作处使用。

**预期效果**：移动/重命名已接入，用户感知延迟降低；其它场景接入 Hook 后交互更流畅。

## 七、渲染优化（间接降低请求压力）

### 7.1 虚拟列表 ✅ 已实现并接入

**原理**：只渲染可视区域 + 少量缓冲区，减少 DOM 节点。

**当前实现**：
- ✅ 组件：`frontend/src/components/files/VirtualizedFileGrid.tsx`，手写实现（无 `@tanstack/react-virtual` 等外部依赖），根据窗口滚动和 `FILE_LIST.VIRTUAL_GRID_ROW_HEIGHT`（360px）计算可视行范围，只渲染 `[startRow - overscan, endRow + overscan]`，响应式列数（2–6 列）；resize 监听 150ms 节流。
- ✅ 文件列表启用条件：`FileList.tsx` 在 `files.length > FILE_LIST.VIRTUAL_THRESHOLD`（24）时启用 `VirtualizedFileGrid`，否则使用 `FileGrid`；列表参与整页滚动（无内嵌滚动容器），避免可视窗口被锁死。

**后续可选方向**：
- 如文件数进一步增大，可继续调优 `VIRTUAL_GRID_ROW_HEIGHT` 与 overscan 策略，或对超大数据集场景做更激进的虚拟化策略。

### 7.2 分页 + 无限滚动 ✅ 已实现

**当前实现**：
- ✅ 后端分页：`FILE_LIST.LIMIT`（100）条/页；`useFileList` 的 `loadFiles`/`loadMore` 使用 `page`、`limit` 参与 `getCacheKey` 与请求。
- ✅ 前端无限滚动：`FileList.tsx` 中 `InfiniteScrollSentinel` 使用 `IntersectionObserver` 监听列表底部，自动触发 `loadMore`（经 400ms 节流）；`FileListPagination` 在传入 `onLoadMore` 时以「已加载 x / y 页 + 加载更多」模式工作。
- ✅ 列表随页面整体滚动（无内嵌滚动容器），避免可视窗口被锁死。

## 八、预加载与预取

### 8.1 列表「下一页」预取 ❌ 未实现

**说明**：
- 当前策略是**按需加载下一页**：后台分页 + 无限滚动 /「加载更多」按钮触发 `loadMore`，不会在用户未触达列表底部时主动预取下一页，以避免在高并发场景下额外占用带宽与后端吞吐。

## 九、请求重试与错误处理

### 9.1 指数退避重试 ✅ 已实现（GET 全局）

**当前实现**：
- ✅ 分片上传：`CHUNKED_UPLOAD.MAX_RETRIES = 5`，大文件分块上传失败会进行有限次重试。
- ✅ 普通 GET 请求：`frontend/src/utils/retry.ts` 提供 `retry(fn, { maxRetries, initialDelay, maxDelay, backoff })` 指数退避；`api.ts` 响应拦截器中对 **GET 请求** 在以下场景自动重试：网络错误（`ERR_NETWORK`）、超时（`ECONNABORTED`）、服务器 5xx。
- ✅ 重试参数：`REQUEST.RETRY_MAX`（3）、`RETRY_INITIAL_DELAY_MS`（1000）、`RETRY_MAX_DELAY_MS`（10000）；重试逻辑在 `api.ts` 的 response 拦截器 `catch` 中调用 `retry(...)`。

**效果**：
- 在后端短暂抖动时，前端无需人工刷新即可自动恢复，减少「瞬时错误」对用户的影响，同时不会对后端造成无限重试压力。

## 十、Service Worker 离线缓存 ✅ 已实现

**原理**：使用 Service Worker 预缓存静态资源，并对文件列表/文件夹 API 做运行时 NetworkFirst 缓存，离线时回退到缓存。

**当前实现**：
- ✅ **vite-plugin-pwa**：`frontend/vite.config.ts` 中配置 `VitePWA`，`registerType: 'autoUpdate'`；仅生产环境（`import.meta.env.PROD`）在 `main.tsx` 中动态 `import('virtual:pwa-register').then(({ registerSW }) => registerSW({ immediate: true }))` 注册 SW。
- ✅ **静态资源预缓存**：`workbox.globPatterns: ['**/*.{js,css,html,ico,png,svg,woff2}']`，构建产出的 JS/CSS/HTML/图标/字体等全部预缓存，离线可加载页面与资源。
- ✅ **文件列表 API 运行时缓存**：`urlPattern: /\/api\/files\?/`，handler `NetworkFirst`，`cacheName: 'file-list-cache'`，`expiration: { maxEntries: 128, maxAgeSeconds: 86400 }`，`networkTimeoutSeconds: 10`；网络超时或失败时回退到缓存。
- ✅ **文件夹 API 运行时缓存**：`urlPattern: /\/api\/folders(\/contents)?(\?|$)/`，handler `NetworkFirst`，`cacheName: 'folders-cache'`，`expiration: { maxEntries: 64, maxAgeSeconds: 86400 }`，`networkTimeoutSeconds: 10`。
- ⚠️ SW 与 localStorage 列表缓存独立：SW 按请求 URL 缓存 GET 响应；localStorage 列表缓存由 `fileListCache` 管理，失效逻辑见 4.3。

**适用场景**：
- 文件列表缓存（离线可查看已访问过的列表与文件夹结构）
- 静态资源缓存（JS/CSS/图片/字体）

**预期效果**：重复访问请求 ↓，弱网/离线体验提升；离线时可查看此前打开过的文件列表与文件夹。

## 📝 注意事项

1. **缓存失效策略**：移动文件/文件夹、批量移动、登出后已调用 `clearFileListCache()`（见 4.3）；删除文件、上传完成后当前依赖 SWR 后台 revalidate 或用户切换条件后新请求，若需「删除/上传后立即让其它页缓存失效」可在对应成功回调中补充 `clearFileListCache()`。Service Worker 运行时缓存与 localStorage 列表缓存独立，SW 按 URL 缓存 GET 响应，过期由 Workbox 的 `maxAgeSeconds`/`maxEntries` 控制。
2. **错误处理**：所有优化手段需有降级方案（如缓存失败不影响主流程、revalidate 失败静默不覆盖已有展示、重试失败后不再无限重试）。
3. **监控**：可添加请求数、去重命中率、列表缓存命中率、SW 缓存命中率、错误率等监控，便于评估与调优。
4. **测试**：高并发与弱网场景下验证请求取消、限流、去重、SWR、SW 离线回退等行为符合预期。
