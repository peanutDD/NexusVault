# 参考 B 站技术升级本项目

> 基于 B 站公开的技术博客与架构实践（CDN、视频传输、限流熔断、微服务等），对照本仓库现状整理可落地的升级项。  
> 相关文档：[高并发.md](./高并发.md)、[前端降低后端并发压力方案.md](./前端降低后端并发压力方案.md)、[backend/docs/BACKEND_IMPROVEMENTS.md](../backend/docs/BACKEND_IMPROVEMENTS.md)。

---

## 一、B 站技术要点摘要

| 领域 | 实践 | 来源 |
|------|------|------|
| **CDN** | 网关（H1/H2/H3）+ 缓存 + 回源 + 中控；一致性哈希负载；预取窗口，请求时延↓40% | 下行 CDN 架构探索与应用 |
| **视频** | DASH + .m4s 分片、MSE、H.264/H.265；直播/点播分离调度 | 视频播放技术、融合资源调度 |
| **调度** | 成本调度层 → 资源调度层 → 业务调度（流名/稿件级）；带宽复用↑、卡顿率↓ | 融合资源调度 |
| **限流/熔断/降级** | 滑动窗口/令牌桶、熔断防雪崩、非核心功能降级 | 微服务与高可用实战 |
| **自适应限流** | BBR：CPU、QPS、RT、inflight 综合判断 | Kratos/Sentinel 系 |
| **高可用** | 多级超时、依赖隔离、StatsD+Grafana、Kafka 异步 | 毛剑 B 站 Go 服务实践 |

## 二、可升级项（按领域）

### 2.1 网关与协议层

| 项目 | 现状 | B 站做法 | 建议 | 优先级 |
|------|------|----------|------|--------|
| **HTTP/2** | Axum 默认可支持，取决于 serve 配置 | 网关支持 H1/H2/H3，多路复用 | 确认 `main` 里 `hyper`/`axum` 是否启用 HTTP/2（若反向代理已终结 TLS 则可在代理层开） | 低 |
| **请求超时分层** | 全局 `TimeoutLayer::new(120s)` | 连接/读/写多级超时 | 对下载/上传大文件路由单独更长超时或仅对 API 短超时；避免所有接口一刀切 120s | 中 |
| **Range / 预取** | 已支持 Range、If-Range、multipart byteranges | 分片 + 预取窗口，时延↓40% | 列表/预览可考虑「预取下一段」策略（前端或后端约定）；当前已具备 Range 基础 | 低 |

**落地建议**：  
- 在 `app.rs` 或路由层区分「普通 API 超时」（如 30s）与「下载/上传流式超时」（120s 或按配置）。  
- HTTP/2：若用 Nginx/Caddy 反向代理，在代理上开启 HTTP/2 即可，后端可暂不改。

### 2.2 视频与流媒体

| 项目 | 现状 | B 站做法 | 建议 | 优先级 |
|------|------|----------|------|--------|
| **大视频预览** | 超过阈值用 HLS（.m3u8 + .ts），前端先走 `/preview` 流式播放，HLS 就绪后无缝切换并保留进度/播放状态/音量 | DASH + .m4s、MSE、多码率 | 保持 HLS 作为主方案，结合「先流式后 HLS」降低首播等待；多码率/自适应可远期考虑 DASH 或 HLS 多档位 | 低 |
| **HLS 生成** | 异步：`/hls/prepare` 触发转码，`/hls/status` 轮询就绪，避免请求阻塞 | 转码异步、预生成/预热 | 保持异步化链路；必要时在上传完成后预热或引入 WebSocket 通知 | 高（已完成） |
| **S3 视频** | 当前 HLS 仅 local，S3 不支持 | 点播走 CDN/对象存储 | S3 场景下：要么预签名直链 + 前端用 hls.js 播原文件（小视频）；要么异步任务把 S3 拉到临时目录再转 HLS（与上条一致） | 中 |

**落地建议**：  
- 已落地：HLS 异步化（prepare/status）+ 前端先流式播放、就绪后切换并保持播放进度、播放状态与音量/静音。  
- 需要时再引入上传后预热或 WebSocket 通知，减少轮询与转码抖动。  
- 与 [BACKEND_IMPROVEMENTS](../backend/docs/BACKEND_IMPROVEMENTS.md) 中「后台任务」思路一致，可复用维护任务那套「任务状态 + 清理」设计。

### 2.3 限流、熔断与降级

| 项目 | 现状 | B 站做法 | 建议 | 优先级 |
|------|------|----------|------|--------|
| **限流算法** | 按 IP 固定窗口（如 500/60s），moka 缓存 | 滑动窗口/令牌桶/BBR | 可引入滑动窗口或令牌桶，减少临界突发；若流量不大可保持现状 | 低 |
| **按用户限流** | 仅按 IP | 按 user_id 更公平 | 认证后按 `user_id` 限流（与 [BACKEND_IMPROVEMENTS](../backend/docs/BACKEND_IMPROVEMENTS.md) 中「按 user_id 限流」一致） | 中 |
| **熔断** | 无 | 下游异常时快速失败、探测恢复 | 对「依赖存储/DB」的接口可加熔断（如连续失败 N 次则短时直接 503），需引入熔断状态机或 crate | 低 |
| **降级** | 仅有 LoadShed（过载 503） | 双 11 等关闭非核心功能 | 可配置「降级开关」：例如关闭预览/列表缓存、仅保留上传下载；与配置中心或环境变量联动 | 低 |
| **全局并发** | `ConcurrencyLimitLayer::new(512)` + `LoadShedLayer` | 合理并发 + 过载保护 | 已对齐「限并发 + 过载即 503」的思路；可把 512 改为配置项便于按机器调优 | 低 |

**落地建议**：  
- 优先做「按 user_id 限流」和「超时分层」，再视需要做滑动窗口/熔断/降级开关。

---

### 2.4 下载与 CDN

| 项目 | 现状 | B 站做法 | 建议 | 优先级 |
|------|------|----------|------|--------|
| **Range / 缓存头** | 已支持 Range、ETag、If-None-Match、If-Modified-Since、If-Range | 分片 + 缓存 | 保持；可再核对 `Cache-Control` 对静态预览/下载的 max-age 是否合理 | 低 |
| **直链/CDN** | 文件经本机读流或 S3 代理 | 点播走 CDN，回源 | 若用 S3：改为「预签名 URL 重定向」直下，本机不扛带宽；若有 CDN 可在前面加一层 CDN 回源 | 高（成本/带宽敏感时） |
| **批量下载** | 已有 ZIP 流式 | 批量打包 + 流式 | 已对齐 | - |

**落地建议**：  
- S3 场景：下载接口优先返回 302 + 预签名 URL，由客户端或 CDN 直连对象存储，与 [高并发.md](./高并发.md) 中「预签名 + 直下」一致。  
- 若有自有或云 CDN，可在 Nginx/网关配置回源到当前下载路径，并设置缓存策略。

### 2.5 可观测与运维

| 项目 | 现状 | B 站做法 | 建议 | 优先级 |
|------|------|----------|------|--------|
| **请求追踪** | `X-Request-ID`、tracing span | trace ID 全链路 | 已具备 request_id；若引入多服务可考虑 W3C TraceContext 或 OpenTelemetry | 低 |
| **监控** | Prometheus /metrics、健康检查 | StatsD + Grafana、告警 | 保持 Prometheus；可增加「按路由/状态码」的 QPS/延迟分桶，便于做 B 站式的「业务调度」观察 | 中 |
| **健康语义** | /health、/livez、/readyz（与 health 同） | ready vs live 区分 | [BACKEND_IMPROVEMENTS](../backend/docs/BACKEND_IMPROVEMENTS.md) 已建议：/readyz 检 DB+存储，/livez 仅进程存活 | 低 |
| **后台任务** | 维护任务仅 log | 连续失败告警 | 对「连续失败次数」打 metric 或告警 | 低 |

**落地建议**：  
- metrics 增加按 `path` 或 `route` 的标签（若尚未有），便于看「列表/下载/上传」各自延迟与错误率。  
- 与现有 BACKEND_IMPROVEMENTS 中「健康检查语义」「后台任务监控」合并排期。

| 项目 | B 站场景 | 本项目对应 | 说明 |
|------|----------|------------|------|

## 三、建议实施顺序

1. **短期（1–2 周）**  
   - **HLS 异步化**：已完成（prepare/status + 轮询），前端先流式播放、就绪后切换。  
   - **超时分层**：API 短超时（如 30s）、下载/上传流式保持 120s 或配置。  
   - **按 user_id 限流**：认证后限流 key 含 user_id，避免单 IP 多账号占满配额。

2. **中期（2–4 周）**  
   - **S3 下载直链**：下载接口返回 302 预签名 URL，减轻本机带宽。  
   - **Metrics 细分**：按路由/状态码分桶，便于做「业务级」监控与调度决策。  
   - **健康检查语义**：/readyz 与 /livez 区分，并写入 BACKEND_IMPROVEMENTS。

3. **长期（按需）**  
   - 滑动窗口/令牌桶限流、熔断、降级开关。  
   - HTTP/2 在反向代理层开启；视频多码率/DASH（若业务需要）。  
   - 全链路 Trace（多服务时再上）。
