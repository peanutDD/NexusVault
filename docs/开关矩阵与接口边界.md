# 开关矩阵与接口边界（本地全量落地 / 云端可开关）

本文件用于：

- 把“高并发全量落地方案”拆成可逐步启用的**功能开关（Feature Flags）**
- 把涉及云端的能力收口为**可替换的 Provider 接口**（本地实现 / 云端实现）
- 确保以后对接云端时，只需要替换 Provider，不推翻现有业务逻辑

> 约定：所有开关默认以“本地可跑通”为第一优先级；云端对接时仅改配置/替换 provider。

---

## 0) 总体原则（高并发下必须遵守）

- **读链路优先缓存**：先降低 DB 压力，再谈分库分表
- **大流量从 API 卸载**：下载/预览/分发尽量走对象存储 + CDN（本地先用同一协议跑通）
- **重活全部异步化**：转码/压缩等 CPU/IO 密集任务必须走队列 + worker
- **每条链路可独立扩容**：API、worker、存储/缓存、DB 都是独立伸缩维度

---

## 1) 开关矩阵（Feature Flags）

| 开关名 | 默认值（本地） | 本地行为（立即可验证） | 云上行为（以后对接） | 主要涉及模块 |
| --- | --- | --- | --- | --- |
| `ZIP_CACHE_ENABLED` | `0` | 批量 ZIP 首次仍走“边打包边传”（首包更快）；开启后才会写入本地缓存目录并在下次命中复用 + Range | 产物写对象存储，配合 CDN 缓存与 Range | 后端：批量 ZIP handler/service |
| `ZIP_CACHE_BACKEND` | `local` | `local`：落盘缓存 | `s3`：落对象存储 | 后端：ZipArtifactStore provider |
| `ZIP_CACHE_TTL_SECS` | `3600` | 本地定时清理过期产物 | 云端按 TTL 或生命周期规则清理 | 后端：zip cache 清理任务 |
| `ZIP_BUILD_MAX_CONCURRENT` | `2` | 限制批量 ZIP 打包并发，保护 CPU/IO，避免影响其他请求 | 同上（可配合队列化/独立 worker） | 后端：批量 ZIP handler/service |
| `CACHE_ENABLED` | `1`（无 Redis 自动降级） | 走 memory/redis cache-aside，缓存元数据与列表第一页 | Redis/ElastiCache；可扩展 tag 失效 | 后端：CacheProvider；前端仍可保留 SWR |
| `CACHE_BACKEND` | `redis` | 有 `REDIS_URL` 才启用 redis，否则降级 `memory` 或关闭 | `redis` | 后端：CacheProvider |
| `CACHE_DEFAULT_TTL_SECS` | `60` | 默认 TTL（列表/元数据等按业务覆写） | 同上 | 后端：CacheProvider |
| `TRANSCODE_MAX_CONCURRENT` | `2` | 限制转码 worker 并发，保护 CPU/IO | GPU/多 worker 时仍保留全局配额 | worker：转码执行器 |
| `TASK_TYPE_CONCURRENCY_gif_preview` | `2` | gif_preview 专用并发上限 | 同上 | worker：任务执行 |
| `TASK_QUEUE_BACKEND` | `postgres` | 使用 `background_tasks` + `FOR UPDATE SKIP LOCKED` | 未来可扩展 `redis/kafka/sqs`（当前仅支持 postgres） | 后端：TaskQueueProvider |
| `DOWNLOAD_MODE` | `proxy` | 下载走 API 代理（现有 Range） | `presigned` / `redirect`（需 `STORAGE_BACKEND=s3`） | 后端：DownloadLinkProvider |
| `PRESIGN_TTL_SECS` | `300` | presigned URL 有效期（秒）；仅 `DOWNLOAD_MODE!=proxy` 时生效 | 云端结合 CDN/对象存储 | 后端：DownloadLinkProvider |
| `ENABLE_HTTP_CACHE_HEADERS` | `1` | 给缩略图/HLS 分片/派生文件加 Cache-Control/ETag | CDN 直接命中缓存 | 后端：response utils / handlers |
| `READ_REPLICA_DATABASE_URL` | 空（禁用） | 不启用读写分离 | 读请求走 replica，写走 primary | 后端：DbRouter/Pool |
| `HLS_ABR_MAX_VARIANTS` | `1` | 默认单档；逐步放开时增加为 2/3/4（生成 master.m3u8 + variants） | 云端可扩展更多档位 + GPU | 后端：HLS 转码 pipeline |
| `HLS_ABR_VARIANTS` | `240:350,360:700,480:1200,720:2500` | 本地先跑少量档位验证产物与切换 | 云端可扩展更多档位 + GPU | 后端：HLS 转码 pipeline |

> 建议：开关由后端 config 统一加载（env + 默认值），并在 `/metrics` 中暴露当前开关状态（label 或 info gauge），避免“线上到底开没开”不可观测。

## 2) Provider 接口边界（只替换实现，不推翻业务）

下面这些接口建议作为“稳定边界”，上层业务只依赖 trait，不依赖具体实现（local/s3、postgres/redis/kafka、proxy/presign）。

### 2.1 ZipArtifactStore（ZIP 产物缓存）

**职责**

- `get(key) -> Option<ArtifactHandle>`：命中则返回可读（支持 Range）的句柄
- `put(key, stream/file) -> ArtifactHandle`：写入产物
- `delete(key)` / `gc(ttl)`：清理过期

**不负责**

- 不负责“哪些文件被打包/权限校验”（这属于业务层）

**实现**

- `LocalZipArtifactStore`：本地目录 + 元数据（mtime/size）
- `S3ZipArtifactStore`：对象存储 key + 可选 CDN

---

### 2.2 CacheProvider（元数据/列表缓存）

**职责**

- `get_json<T>(key)` / `set_json(key, value, ttl)` / `del(key)`
- 允许 tag 或 pattern 失效（可选阶段）

**关键策略**

- cache-aside：读 miss 回源 DB，再写缓存
- 失效：重命名/删除/移动等写操作后，清理相关 key（至少：文件元数据 key、列表第一页 key）
- 防击穿：允许短 TTL + 请求合并（单机内 mutex/once 或 redis lock）

**实现**

- `MemoryCacheProvider`（本地兜底）
- `RedisCacheProvider`（本地/云端通用）

---

### 2.3 TaskQueueProvider（后台任务队列）

**职责**

- `enqueue(task)`：入队
- `dequeue(task_type, lease)`：出队并加 lease
- `ack/succeed/fail/retry`：状态流转（含 attempts、next_run_at）
- `requeue_stuck`：回收卡死任务
- `get_queue_depth`：可观测指标（pending_ready/running/failed）

**实现**

- `PostgresTaskQueueProvider`（已存在）
- `RedisTaskQueueProvider` / `KafkaTaskQueueProvider` / `SqsTaskQueueProvider`（未来对接）

---

### 2.4 DownloadLinkProvider（下载直链/签名）

**职责**

- `resolve_download(file, mode) -> DownloadResolution`
  - `Proxy`：走 API（现有行为）
  - `Redirect(url)`：302 到对象存储/CDN
  - `Presigned(url, headers)`：返回短期有效 URL（必要时带 header 要求）

**实现**

- `LocalDownloadLinkProvider`：Proxy 或直连本地（便于本地验证协议）
- `S3PresignedDownloadLinkProvider`：真正 presign

---

### 2.5 DbRouter（读写分离）

**职责**

- `read_pool()` / `write_pool()`：根据开关选择 pool
- 对外提供“读取路径使用 read pool”的约定

**实现**

- 本地：read=write（同一个 pool）
- 云端：read=replica，write=primary

---

### 2.6 TranscodePipeline（转码流水线：单码率 → ABR）

**职责**

- `ensure_hls_ready(file)`：确保 HLS 产物存在（单码率或多档位）
- `variant_playlist` 生成规则固定、可重入（幂等）
- 对资源配额敏感（遵循 `TRANSCODE_MAX_CONCURRENT` / task_type 并发）

**实现**

- 本地：CPU 转码（少档位）+ 产物落本地目录
- 云端：GPU/集群转码 + 产物落对象存储 + CDN

---

## 3) 与“完全落地顺序”的映射

1. ZIP 产物缓存复用：`ZIP_CACHE_*` + `ZipArtifactStore`
2. Redis cache-aside：`CACHE_*` + `CacheProvider`
3. Cursor 分页：list API 通过 `pagination=cursor` 或 `cursor` 启用（repo 层 keyset）
4. 转码配额 + 多队列隔离：`TRANSCODE_MAX_CONCURRENT` / `TASK_TYPE_CONCURRENCY_*` +（可选）按 task_type 分队列
5. Provider 化：`DOWNLOAD_MODE`、`TASK_QUEUE_BACKEND`、`READ_REPLICA_DATABASE_URL` + 上述 Providers
6. ABR 多码率：`HLS_ABR_MAX_VARIANTS` + `HLS_ABR_VARIANTS` + HLS 转码 pipeline
